# DataWhisperer 面试讲解稿

这份文档帮你在面试时讲清楚项目。不要死背，可以按自己的话表达。

## 30 秒版本

DataWhisperer 是一个自然语言数据分析智能体。用户输入中文问题后，系统会读取 MySQL 数据库结构，调用大模型生成 SQL，再经过服务端 SQL 安全校验，执行查询，最后返回表格、ECharts 图表和业务分析结论。

它不是单纯调一个大模型接口，而是把大模型接入到了一个完整的数据分析工作流里，重点考虑了 schema 理解、SQL 安全、查询执行、图表推荐和结果解释。

## 1 分钟版本

我做的 DataWhisperer 是一个 Text-to-SQL 数据分析智能体，面向不会写 SQL 的业务用户。用户可以输入“最近 6 个月每月销售额趋势”这类中文问题，系统会先读取 MySQL 示例库的表结构，把 schema 压缩成适合 prompt 的文本，然后调用 OpenAI-compatible 大模型生成 SQL。

生成 SQL 后，系统不会直接执行，而是先经过服务端安全校验，只允许 `SELECT` 或 `WITH` 这类只读查询，禁止写入、删除、DDL 和多语句，并自动补充 `LIMIT`。查询执行完成后，系统会根据字段类型推荐 ECharts 图表，再生成简短业务结论，同时把执行轨迹返回给前端。

这个项目第一阶段是单主控工具链，后续可以继续扩展 MCP 工具化、RAG 指标口径库和多智能体协作。

## 面试官可能会问什么？

### 你这个项目和普通 ChatGPT 写 SQL 有什么区别？

普通 ChatGPT 只是生成 SQL，本项目做的是完整闭环：

- 自动读取数据库 schema。
- 根据业务问题生成 SQL。
- 服务端做 SQL 安全校验。
- 执行查询并返回真实结果。
- 自动生成图表和分析结论。
- 返回 trace，方便解释执行过程。

核心区别是：它是一个可运行的数据分析系统，不只是 prompt demo。

### 为什么要把 schema 压缩后再喂给模型？

真实企业数据库表很多、字段很多，如果直接把完整 JSON 全塞进 prompt：

- token 成本会上升。
- 模型容易被无关字段干扰。
- 上下文越长，回答越不稳定。

所以我把 schema 压缩成更接近人类读表结构的文本，只保留表名、字段、类型、主键和外键关系。这样既节省 token，也更利于模型理解。

### 为什么 SQL 安全不能只靠提示词？

因为提示词不是安全边界。模型可能被用户诱导，也可能输出不符合要求的 SQL。

所以服务端必须做硬校验：

- 只允许只读查询。
- 禁止危险关键字。
- 禁止多语句。
- 禁止注释绕过。
- 自动加 `LIMIT`。

这部分是工程安全设计，不应该交给模型自己保证。

### 为什么第一阶段没有直接做多智能体？

因为多智能体会增加调试成本和系统复杂度。第一阶段更重要的是先把真实链路跑通：

```text
自然语言 -> SQL -> 查询结果 -> 图表 -> 分析结论
```

等工具层稳定后，再拆成多个角色才更有意义。否则一开始上多智能体，很容易变成看起来复杂但不可控。

### 这个项目后续怎么扩展 MCP？

当前 Tools 层已经按工具边界拆好了。后续可以把这些能力包装成 MCP 工具：

- 读取 schema
- 生成 SQL
- 校验 SQL
- 执行查询
- 生成图表
- 生成分析结论

主控 Agent 通过 MCP 调用工具，这样可以让工具能力被其他 Agent 或客户端复用。

### 如果大模型生成错 SQL 怎么办？

当前 v1 主要有三层处理：

- SQL 安全校验，保证不会执行危险语句。
- 查询异常捕获，数据库报错时不会让服务崩掉。
- 演示兜底规则，没有模型或模型输出不稳定时，典型问题仍然可以跑通。

后续可以继续加评测集和自动修复链路，比如 SQL 执行失败后，把错误信息反馈给模型进行一次修正。

## 可以重点强调的工程点

- 我把大模型能力放在工作流里，而不是只做聊天。
- 我做了 SQL 安全兜底，知道 prompt 不能当安全边界。
- 我把核心能力拆成 Tools，方便测试和后续 MCP 化。
- 我用了 Pydantic 定义请求和响应，让接口契约清晰。
- 我写了 pytest 和 ruff，保证基本质量。
- 我做了中文控制台，方便非技术用户体验。

## 项目演示顺序

1. 打开控制台：`http://127.0.0.1:8081/`
2. 点击“查询最近 6 个月每月销售额趋势”。
3. 展示分析结论、图表和表格。
4. 切换到 SQL tab，说明模型生成的 SQL。
5. 切换到轨迹 tab，说明 Agent 执行流程。
6. 打开 `/docs`，展示接口文档。
7. 展示 `tests/test_sql_safety.py`，强调安全校验。

## 一句收尾

这个项目的价值在于：它把大模型从“会回答”变成了“能安全地调用工具完成数据分析任务”。
